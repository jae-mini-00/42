🔐 Session Timeout Testing Guide

Overview

The webserver implements two-level timeout:

  • Socket Timeout: Disconnects idle clients after 30 seconds
  • Session Timeout: Invalidates idle authenticated sessions after 30 seconds

This guide demonstrates how to test the session timeout functionality with a real-time visualization.

Method 1: Quick Visual Test (No Setup Required)

1 Open the Session Monitor

Click here to open Session Timeout Monitor

This page will:

  • Display a 30-second countdown timer
  • Show a visual progress bar
  • Change colors as time runs out (green → orange → red)
  • Track when the session expires
2 Watch the timeout in real-time

The countdown will show you:

  • Elapsed time since page load
  • Remaining time before session expires
  • Real-time visual progress (updates 10x per second)
3 Verify timeout behavior
  • After 30 seconds, the timer shows "00:00"
  • The progress bar turns red and displays "SESSION EXPIRING"
  • Status changes to "Session Expired"
  • Refresh the page to start a new session

Method 2: Programmatic Test (Using curl)

1 Start the webserver 
./webserv siyoung.wbsrv
2 Create a session by making a login request 
curl -i -X POST http://localhost:8080/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=test&password=test"

Look for the Set-Cookie: session_id=... header in the response.

3 Monitor server logs

The server will periodically clean up expired sessions:

  • Each event loop iteration calls clean_expired_sessions()
  • Sessions older than 30 seconds are automatically removed
  • No log output by default (silent cleanup)
4 Verify by accessing protected routes 
# Within 30 seconds (should work)
curl -i -H "Cookie: session_id=YOUR_SESSION_ID" \
  http://localhost:8080/storage/

# After 30 seconds (should fail with 401)
sleep 35
curl -i -H "Cookie: session_id=YOUR_SESSION_ID" \
  http://localhost:8080/storage/

Implementation Details

How Session Timeout Works

1. Client creates session → session_id cookie issued
2. Each access → last_access timestamp updated
3. Server loop → clean_expired_sessions() called every iteration
4. Sessions idle >30s → automatically deleted
5. Next access with expired session_id → 401 Unauthorized

Configuration

Timeout value comes from siyoung.wbsrv config:

:8080 =
    ...30            # 30 seconds server response timeout
    GET /storage/ <- /spool/www/storage/
        @ auth_info  # Requires valid session

Code Location

  • src/server/Session.hpp - Session data structure and timeout logic
  • src/server/Session.cpp - clean_expired_sessions() implementation
  • src/server/Server.cpp - Server loop calls cleanup (line ~365)

Testing Checklist

  • Open /session-info.html and watch the 30-second countdown
  • Refresh page before timeout → new 30-second countdown starts
  • Wait 30+ seconds without interaction → session expires and shows "expired"
  • Try accessing protected routes after timeout → should be denied
  • Socket disconnects after 30 seconds of inactivity (check with nc or telnet)
💡 Tip: The session-info page uses browser localStorage, so login time persists even if you refresh. This accurately simulates a real session where the server tracks activity independently.
⚠️ Note: Both socket and session timeouts use the same 30-second value from config. In production, you might want different values (e.g., 30s socket timeout, 15m session timeout).
